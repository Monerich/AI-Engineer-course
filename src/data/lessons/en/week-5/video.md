# Сценарий видео: Advanced n8n: Sub-workflows and Data Arrays

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Scene 1:** Speaker close-up, transitioning to a dark-mode n8n canvas showing JSON data flowing between two nodes. The text "Array of JSON Objects" highlights on screen. | Welcome to Week 5. In n8n, all data inputs and outputs are structured strictly as an array of JSON objects. By default, if a node receives multiple items, it implicitly iterates and runs for each item. Remember, execution order is deterministic: parallel branches always execute from topmost to bottommost, and leftmost to rightmost. |
| **Scene 2:** Screen recording of the n8n canvas. A 'Loop Over Items' node is added. The 'loop' branch is wired to an HTTP request, and the output wires back into the Loop node. | To process massive arrays without crashing, we use the Loop Over Items node, formerly known as Split in Batches. It features a 'loop' route that passes a specific batch size forward, and a 'done' route. You must physically wire the end of your loop sequence back into the input of this node to complete the cycle. |
| **Scene 3:** A network tree diagram appears. A large "Parent" workflow box points down to three isolated "Child" workflow boxes. | As your automations grow, visual spaghetti becomes unmaintainable. The Execute Workflow node allows you to call one workflow from another. This modular architecture extracts complex logic into dedicated sub-workflows, which always begin with an Execute Sub-workflow Trigger. |
| **Scene 4:** Split screen. On the left, a Parent node configuring 'Define using Fields below'. On the right, the Child workflow receiving the JSON payload. | Variable isolation is critical for security and stability. Child workflow variables are strictly isolated from the parent. To pass data, you explicitly map expressions in the parent using the 'Define using Fields below' setting. The child workflow receives only these exact keys as its initial payload. |
| **Scene 5:** High-speed screencast building a flow: RSS Read -> Split Out/Limit -> Loop Over Items -> OpenAI -> Aggregate. | Let's practice with a News Digest. We ingest an RSS feed, but to save LLM tokens, we immediately use a Limit or Split Out node to restrict the array to the top 5 items. We pass these into a Loop node, summarize each article using OpenAI, and finally pull the array back together using an Aggregate node. |
| **Scene 6:** Architectural diagram: n8n Main Instance -> Redis Queue -> Worker Node 1, Worker Node 2, Worker Node 3. | When running thousands of parallel executions, standard Node.js heap memory will crash. Enterprise self-hosted n8n environments solve this using Queue Mode. The main n8n instance pushes executions to a Redis queue, and dedicated background worker processes pull and execute the tasks, allowing infinite horizontal scaling. |
| **Scene 7:** Close-up of the n8n Code node editor set to Python. The code `_input.all()` is highlighted in neon green. | While visual nodes handle routing, complex ETL operations belong in code. By setting the Code node to "Run Once for All Items" using Python, you can access the entire incoming JSON array via the `_input.all()` method. This allows you to programmatically sanitize and map chaotic data before it reaches your LLM. |
| **Scene 8:** Webhook node settings. The cursor changes the "Respond" setting to "Immediately". A DAG routes the data forward. | Production systems are event-driven, relying on Webhooks rather than polling. However, if your AI workflow takes 30 seconds to run, the external system sending the webhook might time out. You must configure your Webhook node to return an HTTP 200 OK response immediately, while the cognitive loop processes asynchronously in the background. |
| **Scene 9:** A Loop node connects to a Wait node (set to 5 seconds), followed by an HTTP Request node with "Retry On Fail" toggled on. | Firing 500 parallel API requests will instantly exhaust token buckets and trigger HTTP 429 lockouts. You must throttle task queues by placing a Wait node inside your loop. Additionally, enable 'Retry On Fail' with exponential backoff on HTTP nodes to gracefully handle temporary network rejections. |
| **Scene 10:** Split screen: A Postgres node executing an UPSERT next to an Error Trigger node capturing a crash traceback. | Finally, volatile memory is dangerous for long-running agents. If a container crashes, data is permanently lost. You must implement distributed state control by upserting execution states to an external database like Postgres after every step. If a failure occurs, a global Error Trigger workflow catches the crash and logs it, allowing safe resumption. |
