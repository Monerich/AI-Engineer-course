# Сценарий видео: Deployment and Durable Execution

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Scene 1:** Speaker close-up, transitioning to an animated Docker container wrapping Python scripts, models, and dependencies. | Welcome to Week 22: Deployment and Durable Execution. Today, we move your AI agents from local scripts to production-grade systems. We start with Docker. Without Docker, your AI application might run perfectly on your laptop but fail entirely on a coworker's machine or in the cloud. We use it to package your code, dependencies, and models into a single, reproducible container. |
| **Scene 2:** Split screen showing a `docker-compose.yml` file on the left and a network topology diagram (FastAPI, Postgres, Redis) on the right. | An enterprise AI application is rarely just one script. In Block 2, we use Docker Compose to orchestrate multiple containers at once. This allows us to securely network our Python backend, a PostgreSQL database for state management, and a message queue together in a local environment before pushing to production. |
| **Scene 3:** Four-column grid displaying logos for Railway, Render, AWS, and GCP, alongside hidden environment variable keys. | Block 3 is about taking your containers live. For rapid prototyping, platforms like Railway offer one-click deployments. For massive scale, deploying on AWS or GCP is like opening a global restaurant franchise, allowing thousands of users to access your agents simultaneously. Crucially, API keys are always injected securely as environment variables. |
| **Scene 4:** Dynamic line chart showing a sudden spike in HTTP requests, triggering new worker nodes to spin up horizontally. | In Block 4, we tackle traffic spikes. An AI system that works for ten users will crash under the weight of five thousand. We implement horizontal autoscaling rules and queue-mode architectures to buffer sudden bursts of tasks, deploying duplicate stateless worker nodes automatically when load increases. |
| **Scene 5:** A traffic routing gateway diverting a request to a Redis cache icon, bypassing the LLM entirely. | Block 5 focuses on efficiency. If thousands of users ask the same question, querying the LLM every time wastes money. We deploy Redis to cache frequent responses. Furthermore, by aggressively utilizing prompt caching on system instructions, we can save up to 90 percent on API costs. |
| **Scene 6:** Enterprise server architecture diagram highlighting a "Kill Switch" and "Sentry" monitoring alerts. | High Availability is the focus of Block 6. We architect resilient servers by integrating tools like Sentry to catch runtime crashes. For deep agent pipelines, we also implement circuit-breakers and auto-rollbacks to isolate and revert serious failures instantly. |
| **Scene 7:** Clean Python code block highlighting `async def`, `await`, and FastAPI endpoint routing. | Block 7 bridges AI and traditional backend engineering. We design real backends using FastAPI and Pydantic to ensure strict data validation. By utilizing asynchronous endpoints and background jobs, we prevent long-running AI requests from blocking the main server thread. |
| **Scene 8:** LangGraph diagram showing a crashed node instantly recovering its state from a PostgreSQL database. | In Block 8, we confront hardware crashes. Any agent running longer than 60 seconds requires durable execution. By persisting the agent's state to a database after every node—using tools like LangGraph's PostgresSaver—we guarantee that an out-of-memory container crash is completely reversible. |
| **Scene 9:** Temporal and Inngest SDK logos overlaying a workflow separating deterministic logic from chaotic tool calls. | Block 9 introduces external orchestration. Inngest offers the easiest path to durability for Python harnesses, while the Temporal SDK turns every individual tool call into a guaranteed durable step. These orchestrators manage exponential backoffs and retries automatically when external APIs fail. |
| **Scene 10:** A locking mechanism on a database preventing a duplicate "charge credit card" API request from executing twice. | Finally, in Block 10, we implement Idempotency laws. When a crashed container restarts and replays its event history, we must prevent it from accidentally repeating irreversible actions. We engineer idempotency keys per step to safely manage in-flight tool calls during a kill, ensuring actions execute exactly once. |
