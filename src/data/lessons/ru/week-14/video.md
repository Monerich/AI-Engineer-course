# Сценарий видео: Продвинутые команды и кастомные инструменты в CrewAI

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Сцена 1:** Крупный план спикера, рядом появляется схема безопасной обвязки базы данных (Safe DB Harness). | Добро пожаловать на четырнадцатую неделю. Мы переходим к продвинутым интеграциям, и первая из них — базы данных. Давать агенту прямой доступ к SQL — это риск. Мы упаковываем аналитические запросы в безопасные инструменты-обвязки, которые блокируют деструктивные команды и автоматически добавляют лимиты, чтобы защитить окно контекста от переполнения. |
| **Сцена 2:** Скринкаст n8n с узлами HubSpot, фокус на скрытом блоке с API-ключами. | Для работы с внешними CRM-системами мы строим семантический слой API. Вместо того чтобы заставлять агента писать сырые HTTP-запросы, мы создаем строго типизированные инструменты. Это изолирует API-ключи от языковой модели и сжимает огромные JSON-ответы CRM в короткие резюме, предотвращая засорение памяти агента. |
| **Сцена 3:** Анимация преобразования сырого текста от Агента-Рисерчера в строгую Pydantic-схему для Агента-Аналитика. | Чтобы агенты не галлюцинировали при передаче данных друг другу, мы внедряем продвинутый формат обмена. Мы обязываем агентов формировать промежуточные результаты в виде строгих JSON-схем. Это правило "чистой передачи" гарантирует, что следующий агент получит идеально структурированные данные. |
| **Сцена 4:** Схема балансировщика нагрузки, защищающего целевой сервер от "стада" суб-агентов. | При параллельном запуске десятков суб-агентов вы мгновенно получите ошибку 429: Too Many Requests от внешних API. Чтобы защитить инфраструктуру от блокировок, мы управляем нагрузкой через асинхронные слушатели, внедряем очереди задач, батчинг и механизмы повторных попыток с экспоненциальной задержкой. |
| **Сцена 5:** Дашборд с графиком падения затрат на токены. Иконки Claude 3.5 Opus и GPT-4o-mini. | Мультиагентные системы — это дорого из-за постоянной пересылки контекста. Экономика CrewAI требует умной маршрутизации. Мы используем дорогие модели только для Оркестратора, отдаем рутину дешевым быстрым моделям и агрессивно кэшируем системные промпты, снижая стоимость одного запуска до 90%. |
| **Сцена 6:** Визуализация папок: `raw/` превращается в `outputs/` с готовым PDF-отчетом. | В финале работы команда должна выдать осязаемый артефакт. Мы настраиваем скрипты для экспорта результатов из внутреннего Markdown в готовые для клиента HTML-страницы или PDF-отчеты, строго отделяя сырые данные от финальной презентации. |
| **Сцена 7:** Значок чекпоинта сохраняется в логотип PostgreSQL после выполнения задачи агентом. | Облачные контейнеры часто перезагружаются. Чтобы избежать полной амнезии мультиагентной системы, мы подключаем внешние базы данных — PostgreSQL или SQLite. Это позволяет сохранять состояние графа после каждого узла, обеспечивая надежное выполнение с возможностью паузы и перемотки. |
| **Сцена 8:** Два агента (Оптимист и Пессимист) отправляют данные Агенту-Судье для вынесения вердикта. | Противоречия между агентами — это инструмент повышения качества. Если один агент находит плюсы, а другой — минусы, мы не усредняем их ответы. Мы используем паттерн "Дебаты" и передаем логи Агенту-Судье, который детерминированно разрешает конфликт и синтезирует глубокий, выверенный ответ. |
| **Сцена 9:** Конвейер из трех изолированных блоков: Автор -> Верификатор с нулевым контекстом -> Редактор. | ИИ-модели слепы к собственным ошибкам. Для финальных текстов и кода мы внедряем цепочки контроля качества. Автор пишет черновик, Верификатор с абсолютно чистым контекстом безжалостно его критикует, а независимый Редактор вносит правки. Это полностью исключает предвзятость модели. |
| **Сцена 10:** CI/CD пайплайн в GitHub Actions блокирует "плохой" пулл-реквест. График Pass Rate. | И наконец, как доказать, что ваша система работает надежно? Мы внедряем автоматическую оценку (Evals). Используя золотые датасеты и LLM-Судью, мы настраиваем CI/CD гейты, которые автоматически блокируют деплой, если обновление промпта или инструмента снижает качество работы роя. |
