# Сценарий видео: Кейс-проект: Локальный CLI Агент с инструментами

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Сцена 1. Ведущий в кадре, рядом появляется схема: LLM + Инструменты (Поиск, Чтение, Запись).** | Приветствую на финальной, двенадцатой неделе. Сегодня мы собираем локального CLI-агента продакшн-уровня. В первом блоке мы создаем интегрированный инструментарий: скрейперы, утилиты для работы с файлами и поисковые API. Именно правильные описания этих инструментов служат строгими инструкциями, чтобы LLM выбирала их без ошибок. |
| **Сцена 2. Анимация: Терминал передает JSON-диалог в базу данных SQLite.** | Блок два посвящен памяти. Относитесь к агенту как к гениальному инженеру с амнезией. Мы реализуем сессионное хранение истории через SQLite, а также применяем метод Карпатого — используем Markdown-файлы, такие как, для экстернализации долговременной памяти. |
| **Сцена 3. Темный экран терминала раскрашивается ANSI-цветами (синий для мыслей, зеленый для действий).** | В третьем блоке мы делаем рантайм агента наблюдаемым. Без логирования выполнения оценки превращаются в субъективные суждения, а ошибки — в слепое блуждание. Мы настраиваем красивое форматирование шагов ReAct-цикла, чтобы визуально разделить внутренний монолог модели и сырые результаты из физического мира. |
| **Сцена 4. Схема: Папка на компьютере -> Значок "глаза" (Watcher) -> Пробуждение агента.** | Блок четыре переводит нас от реактивных скриптов к фоновым агентам. Мы пишем мониторинг директорий. Скрипт на Python перехватывает обновления файлов на уровне ОС. Чтобы не сжечь API-бюджет из-за системного "дребезга", мы обязательно внедряем паттерн Debounce для отложенного запуска. |
| **Сцена 5. Агент завершает работу, на экране появляется автоматически сгенерированный PDF-отчет.** | Пятый блок — это автодокументирование. Согласно правилам инженерии обвязок, каждая сессия обязана оставлять чистое состояние. Перед завершением процесса изолированный Writer-агент компилирует сырые логи SQLite в читаемый отчет по решенным задачам. |
| **Сцена 6. Скринкаст: IDE (VS Code) с кодом на Python (~1500 строк).** | В шестом блоке мы объединяем все концепции на практике. Мы собираем собственную мини-обвязку примерно на 1500 строк Python. Это полноценный локальный ассистент с файловыми инструментами и загрузчиком системного промпта из файла. |
| **Сцена 7. Терминал преображается: появляются Markdown-панели, таблицы и спиннеры загрузки.** | Седьмой блок погружает нас в Python-разработку пользовательского интерфейса. Мы интегрируем библиотеку `rich`. Сырые ANSI-логи заменяются премиальным оформлением терминала: динамическими спиннерами для долгих задач и панелями с подсветкой синтаксиса для аргументов инструментов. |
| **Сцена 8. Схема стриминга: из общего потока текста вырезаются теги `<thought>`.** | Блок восемь решает проблему когнитивной перегрузки пользователя. Чтобы LLM хорошо решала сложные задачи, ей нужна прекогнициция — возможность "подумать". Мы настраиваем скрытие внутренних размышлений внутри тегов `<thought>` с помощью потокового парсера, оставляя для клиента только отполированный итоговый ответ. |
| **Сцена 9. Шкала заполненности контекста доходит до 85%, затем сжимается в один блок.** | В девятом блоке мы спасаем бюджет от переполнения контекстного окна. Мы внедряем алгоритмы сжатия (Context Compaction) при достижении 85% лимита. Для тяжелых задач применяется Filesystem Offload: результаты инструментов свыше 20 тысяч токенов сбрасываются на жесткий диск, а в контексте остается лишь короткий указатель. |
| **Сцена 10. Аварийное завершение скрипта (Ctrl+C), затем команда `--resume` мгновенно возвращает агента к работе.** | Десятый блок делает нашего помощника бессмертным. Мы настраиваем сохранение, загрузку сессий и восстановление состояния. Благодаря Durable Resume, мы транзакционно фиксируем каждый шаг агента в SQLite. После сбоя мы просто загружаем чекпоинт, и агент продолжает работу с того же места. |
