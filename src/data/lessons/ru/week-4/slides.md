# Презентация: Кейс-проект: Умный Telegram-фильтр и лид-парсер

📊 Слайд 1. Блок 1: Интеграция ИИ-нод в n8n
* **Visual Layout Concept:** Тёмная тема (Slate HSL: 220, 10%, 15%), иконки Lucide `Bot`, `Cpu`, `Network`.
* **Ключевые пункты:**
 * Платформа n8n предлагает встроенные ноды для OpenAI, Anthropic и готовый узел AI Agent.
 * Для 90% задач по интеграции ИИ не требуется писать код.
 * Данные из предыдущих шагов передаются в промпт динамически через встроенные выражения.

---
📊 Слайд 2. Блок 2: Парсинг писем с помощью LLM
* **Visual Layout Concept:** Светлая тема (Blue HSL: 210, 20%, 98%), иконки Lucide `FileJson`, `Mail`, `Filter`.
* **Ключевые пункты:**
 * Запрос строгого структурированного вывода (JSON) делает языковые модели пригодными для детерминированных автоматизаций.
 * В узлах OpenAI используется функция `Structured Output Parser` для гарантии соблюдения схемы данных.
 * Извлеченные данные легко маршрутизируются в CRM или таблицы.

---
📊 Слайд 3. Блок 3: Настройка Telegram Bot API
* **Visual Layout Concept:** Тёмная тема (Deep Blue HSL: 220, 50%, 20%), иконки Lucide `MessageSquare`, `Webhook`, `Key`.
* **Ключевые пункты:**
 * Регистрация осуществляется через BotFather с получением безопасного API-токена.
 * Связь с n8n настраивается через узел `Telegram Trigger`, который слушает входящие сообщения.
 * Бот может возвращать Markdown-форматированные уведомления пользователям.

---
📊 Слайд 4. Блок 4: Практикум: Сборка умного Telegram-фильтра лидов
* **Visual Layout Concept:** Светлая тема (Emerald HSL: 150, 30%, 95%), иконки Lucide `UserCheck`, `BellRing`, `GitBranch`.
* **Ключевые пункты:**
 * Базовый скелет: Триггер (новое сообщение) → ИИ-решение (оценка) → Действие (создание лида) → Вывод (алерт).
 * Фильтр классифицирует входящие заявки на «срочно», «обычно» или «спам».
 * Успешная квалификация сокращает время отклика менеджера до секунд.

---
📊 Слайд 5. Блок 5: Интерактивная отладка n8n
* **Visual Layout Concept:** Тёмная тема (Charcoal HSL: 0, 0%, 15%), иконки Lucide `Bug`, `Activity`, `Terminal`.
* **Ключевые пункты:**
 * Визуальный редактор отображает результаты и ошибки выполнения прямо на каждом шаге (узле).
 * Раздел Execution Logs позволяет детально расследовать активные выполнения и сбои API.
 * Централизованный обработчик (Error Trigger) перехватывает падения для быстрой реакции.

---
📊 Слайд 6. Блок 6: Экономика API-вызовов
* **Visual Layout Concept:** Светлая тема (Green HSL: 120, 20%, 95%), иконки Lucide `Banknote`, `TrendingDown`, `Calculator`.
* **Ключевые пункты:**
 * Автоматизация требует просчета юнит-экономики: вызовы дорогих моделей на 1000 писем в день могут сжечь бюджет.
 * Оптимизация достигается кэшированием промптов и правильным дизайном контекста.
 * Для массовых не-real-time задач используется Batch API, дающий скидку 50%.

---
📊 Слайд 7. Блок 7: Асинхронное программирование в Python
* **Visual Layout Concept:** Тёмная тема (Yellow/Black HSL: 45, 90%, 10%), иконки Lucide `FastForward`, `Code`, `Layers`.
* **Ключевые пункты:**
 * Переход от блокирующей библиотеки `requests` к асинхронной `httpx`.
 * Использование Event Loop и `asyncio.gather` для параллельного выполнения API-запросов (Fan-out).
 * Параллелизация радикально снижает общее время выполнения (latency) сложных ИИ-пайплайнов.

---
📊 Слайд 8. Блок 8: Маршрутизация моделей (Cost-Performance Routing)
* **Visual Layout Concept:** Светлая тема (Purple HSL: 260, 30%, 95%), иконки Lucide `GitMerge`, `Scale`, `Brain`.
* **Ключевые пункты:**
 * Применение правила маршрутизации по сложности для жесткого контроля бюджета.
 * Дешевые модели (Haiku 4.5 / GPT-4o-mini) используются для базовой классификации и скрапинга.
 * Мощные модели (Sonnet 4.6 / Opus) применяются только для сложного планирования.

---
📊 Слайд 9. Блок 9: Проектирование нод-заглушек (fallback) и ретраев
* **Visual Layout Concept:** Тёмная тема (Orange HSL: 30, 40%, 15%), иконки Lucide `ShieldAlert`, `Timer`, `LifeBuoy`.
* **Ключевые пункты:**
 * Внедрение экспоненциальной задержки (Exponential Backoff) для защиты от лимитов HTTP 429.
 * Использование встроенных настроек «Retry On Fail» и «Continue On Fail».
 * Переключение на резервные LLM (заглушки) предотвращает полное падение автоматизации.

---
📊 Слайд 10. Блок 10: Сквозная логика рассуждений при квалификации лидов
* **Visual Layout Concept:** Светлая тема (Indigo HSL: 240, 20%, 95%), иконки Lucide `BrainCircuit`, `ListChecks`, `CheckCircle`.
* **Ключевые пункты:**
 * Замена промпт-инжиниринга на контекст-инжиниринг: формирование строгого контекста перед вызовом ИИ.
 * Применение паттерна Precognition (Chain-of-Thought) заставляет модель аргументировать скоринг.
 * Суждение о завершении (оценка лида) экстернализируется через детерминированные узлы n8n.
