# Сценарий видео: Кейс-проект: Умный Telegram-фильтр и лид-парсер

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Сцена 1: Ведущий в кадре, затем плавный переход к интерфейсу n8n с добавлением узлов OpenAI и Anthropic.** | Добро пожаловать на четвертую неделю. Платформа n8n предлагает встроенные узлы для OpenAI и Anthropic, что позволяет настраивать 90 процентов ИИ-интеграций без написания кода. Данные из предыдущих шагов передаются в промпт динамически через встроенные выражения. |
| **Сцена 2: Экран n8n с настройкой узла OpenAI и активацией Structured Output Parser.** | Чтобы языковые модели работали в детерминированных автоматизациях, мы требуем от них строго структурированный JSON-вывод. В узлах OpenAI мы используем функцию Structured Output Parser, которая гарантирует соблюдение заданной схемы данных. |
| **Сцена 3: Запись экрана с интерфейсом BotFather в Telegram, копирование токена и вставка его в n8n.** | Для связи с Telegram мы регистрируем бота через BotFather и получаем безопасный токен. Узел Telegram Trigger настраивается на прослушивание вебхуков, а сам бот может отправлять клиентам уведомления с форматированием Markdown. |
| **Сцена 4: Визуальная схема воркфлоу фильтра лидов: Триггер -> ИИ-решение -> Действие -> Вывод.** | Базовый скелет нашего фильтра лидов прост: Триггер, ИИ-решение, Действие и Вывод. ИИ классифицирует входящие заявки на «срочно» или «спам», что в реальных бизнес-кейсах сокращает время реакции с часов до 30 секунд. |
| **Сцена 5: Демонстрация раздела Execution Logs в n8n и узла перехвата ошибок.** | Платформа n8n позволяет проводить интерактивную отладку: результаты и ошибки отображаются прямо на каждом узле в редакторе. Для расследования сбоев API мы используем раздел Execution Logs и настраиваем глобальный обработчик ошибок. |
| **Сцена 6: Инфографика с расчетом стоимости токенов и иконками кэширования.** | Экономика API-вызовов критична, так как тысячи запросов к дорогим моделям мгновенно сожгут бюджет. Для оптимизации мы агрессивно применяем кэширование промптов, экономящее до 90 процентов средств на префиксах, а некритичные задачи отправляем через Batch API. |
| **Сцена 7: Блок кода на Python, демонстрирующий `import asyncio` и `httpx.AsyncClient`.** | Когда требуются сложные кастомные вызовы, мы переходим на асинхронный Python, заменяя блокирующие запросы на библиотеку httpx. Использование asyncio позволяет распараллелить вызовы API, радикально снижая общую задержку системы при пиковых нагрузках. |
| **Сцена 8: Узел Switch в n8n, направляющий поток к моделям Haiku или Sonnet.** | Маршрутизация моделей или Cost-Performance Routing защищает нашу юнит-экономику. Простые задачи классификации направляются на дешевые модели, такие как Haiku, тогда как мощные модели вроде Sonnet применяются только для сложного рассуждения. |
| **Сцена 9: Настройки узла HTTP Request с включенной опцией Exponential Backoff и fallback-узлом.** | В продакшене API неизбежно падают. Мы внедряем отказоустойчивость: используем ретраи с экспоненциальной задержкой для обхода HTTP 429, и проектируем ноды-заглушки для переключения на резервные LLM при глобальных сбоях серверов. |
| **Сцена 10: Ведущий в кадре, на экране JSON-схема с полем `chain_of_thought`.** | Контекст-инжиниринг заменил классический промптинг. Мы используем паттерн Precognition, заставляя модель мыслить по шагам через поле `chain_of_thought` в JSON, но окончательное суждение о завершении задачи всегда выносит детерминированный узел n8n, а не сам ИИ. |
