# Презентация: Ландшафт AI и основы веб-технологий

📊 Слайд 1. Блок 1 (ИИ-Инженер / Автоматизация): Инфраструктура REST API — настройка GET/POST/PUT/DELETE, Headers, Query/Body параметров в клиенте.
**Визуальный концепт:** Тёмная тема (HSL: 220, 10%, 15%), неоново-синие акценты. Иконки Lucide: `Network`, `ArrowRightLeft`. В центре — схема клиент-серверного взаимодействия.
**Ключевые тезисы:**
* REST API связывает системы: базовое понимание методов заменяет необходимость писать код с нуля.
* **Методы запросов:** `GET` (чтение данных), `POST` (создание новой записи), `PUT` (обновление), `DELETE` (удаление).
* **Параметры:** Query-параметры используются для фильтрации в URL (например, `?status=active`), Body передает основную полезную нагрузку (обычно в JSON).
* **Headers (заголовки):** передают метаданные, такие как тип контента (`Content-Type: application/json`) и токены авторизации.

📊 Слайд 2. Блок 2 (ИИ-Инженер / Автоматизация): Коды ответов и Rate Limiting — семантика HTTP-статусов (2xx, 3xx, 4xx, 5xx) и заголовков Retry-After.
**Визуальный концепт:** Светофорная цветовая схема (Зеленый = 2xx, Желтый = 3xx, Красный = 4xx/5xx). Иконки Lucide: `AlertTriangle`, `Activity`.
**Ключевые тезисы:**
* Чтение кодов ответа — критический навык для отладки пайплайнов.
* **2xx (Успех):** `200 OK` — запрос выполнен успешно.
* **4xx (Ошибка клиента):** `401 Unauthorized` (ошибка ключа), `404 Not Found` (неверный URL), `429 Too Many Requests` (превышен лимит API).
* **5xx (Ошибка сервера):** `500 Internal Server Error` — проблема на стороне сервера.
* **Rate Limiting:** Использование заголовкоВ источниках для настройки пауз (Exponential Backoff) перед повторным запросом.

📊 Слайд 3. Блок 3 (ИИ-Инженер / Автоматизация): Форматирование данных JSON — вложенные объекты, массивы и валидация JSON-схем.
**Визуальный концепт:** Стиль редактора кода (шрифт Monospace, тема Monokai). Иконки Lucide: `Braces`, `FileJson`. Древовидная ASCII-схема JSON-объекта.
**Ключевые тезисы:**
* JSON (JavaScript Object Notation) — универсальный формат обмена данными и "родной язык" для LLM. Вы должны понимать его как словарь, а не как код.
* **Структура:** состоит из пар «ключ-значение», заключенных в фигурные скобки `{}`.
* **Массивы:** упорядоченные списки элементов в квадратных скобках `[]`. В n8n все данные обрабатываются как массивы объектов.
* **Валидация:** Использование JSON Schema для строгого контроля формата вывода LLM, что предотвращает поломку автоматизаций.

📊 Слайд 4. Блок 4 (ИИ-Инженер / Автоматизация): Событийное управление через Webhooks — настройка живого Webhook-слушателя и обработка входящих POST-запросов.
**Визуальный концепт:** Анимация входящего сигнала. Цветовая палитра: глубокий фиолетовый (HSL: 260, 60%, 20%). Иконки Lucide: `Zap`, `Webhook`.
**Ключевые тезисы:**
* Инверсия контроля: вместо постоянного опроса сервера (Polling), внешняя система сама «толкает» данные к вам при наступлении события.
* **Слушатель (Listener):** уникальный URL, который ожидает входящие `POST`-запросы (например, из CRM или Tilda).
* **Timeout Drop:** необходимость мгновенного возврата статуса `200 OK` перед запуском тяжелых ИИ-вычислений для предотвращения дублирования вебхуков.
* Извлечение данных из `Body` входящего вебхука для использования в рабочем процессе.

📊 Слайд 5. Блок 5 (ИИ-Инженер / Автоматизация): Практикум: Работа с публичными API — сборка сквозного конвейера импорта данных Weather/GitHub API.
**Визуальный концепт:** Диаграмма направленного ациклического графа (DAG). Светлая тема для контраста. Иконки Lucide: `CloudSun`, `Github`, `GitMerge`.
**Ключевые тезисы:**
* **Паттерн Orchestrator-Worker:** скрипт собирает данные через HTTP-запросы и передает их в LLM только для синтеза, минимизируя галлюцинации.
* **Базовый скелет:** Триггер (расписание/cron) -> Извлечение данных (GitHub/Weather) -> AI-решение (синтез) -> Действие (отправка в мессенджер).
* Настройка Query-параметров для фильтрации API-ответов и защита от переполнения контекста.
* Объединение нескольких JSON-потоков данных перед отправкой в языковую модель.

📊 Слайд 6. Блок 6 (ИИ-Инженер / Автоматизация): Безопасность и авторизация в веб-интеграциях — настройка Bearer-токенов, Basic Auth и концепт OAuth2.
**Визуальный концепт:** Эстетика защищенного хранилища (сейфа). Иконки Lucide: `Lock`, `ShieldCheck`, `Key`.
**Ключевые тезисы:**
* Поскольку HTTP не сохраняет состояние (stateless), каждый запрос должен содержать авторизацию.
* **Basic Auth:** передача связки логин/пароль в Base64 (устаревший метод).
* **Bearer Tokens (API Keys):** передача криптографического ключа в заголовке `Authorization`. Стандарт для межсерверной M2M коммуникации.
* **OAuth2:** сложный механизм делегированного доступа. Использует Access и Refresh токены для выполнения действий от лица пользователя. Безопасное хранение ключей вне контекста LLM.

**Визуальный концепт:** Имитация окна терминала. Иконки Lucide: `Terminal`, `Box`.
**Ключевые тезисы:**
* **Изоляция зависимостей:** использование `uv` для мгновенного создания чистых виртуальных окружений, предотвращающих конфликты библиотек.
* **Базовый синтаксис:** переменные, условия (`if/else`), циклы (`for/while`), функции.
* **Списки и словари (`list`, `dict`):** эквиваленты массивов и объектов JSON в Python. Критически важны для форматирования системных промптов и API-ответов.
* Обработка исключений (`try/except`) для перехвата сетевых ошибок и ошибок парсинга.

📊 Слайд 8. Блок 8 (Создатель ИИ-Агентов / Агенты & Harness): Введение в дополненные LLM (Augmented LLMs) и базовый цикл рассуждений (Reasoning Loop).
**Визуальный концепт:** Замкнутый цикл с подсвеченными фазами. Иконки Lucide: `BrainCircuit`, `RefreshCw`.
**Ключевые тезисы:**
* **Augmented LLM:** сама по себе модель изолирована. Агент = модель + инструменты (Tools) + память (Memory) + поиск (Retrieval).
* **Цикл рассуждений (ReAct):** Observe (чтение контекста), Think (рассуждение о следующем шаге), Act (вызов инструмента), Evaluate (оценка результата).
* Управление циклом (`while loop`): агент повторяет шаги, пока не достигнет финальной цели, после чего останавливается.
* Важность описания инструментов: модель опирается на описания (schemas) для правильного выбора действия.

---

📊 Слайд 9. Блок 9 (Создатель ИИ-Агентов / Агенты & Harness): Линейные процессы (цепочки шагов) vs Автономные агенты (динамические циклы).
**Визуальный концепт:** Разделенный экран. Слева — линейная блок-схема, справа — динамический граф. Иконки Lucide: `Link`, `Cpu`.
**Ключевые тезисы:**
* **Ключевое отличие:** контроль потока выполнения. В рабочих процессах (workflows) шаги жестко заданы инженером. В агентах — поток динамически определяется самой LLM.
* **Линейные паттерны:** Prompt chaining (цепочки промптов), Routing (маршрутизация), Parallelization (распараллеливание).
* **Когда использовать Agents:** для открытых исследовательских задач с неизвестным числом шагов.
* Преимущества Workflows: предсказуемость, низкая стоимость (агенты потребляют до 15x больше токенов), простота отладки.

---

📊 Слайд 10. Блок 10 (Создатель ИИ-Агентов / Агенты & Harness): Концепция тестового стенда (Agent Harness) как операционной системы контроля ИИ.
**Визуальный концепт:** Схема слоев ОС (Ядро, Память, Приложения). Цвета: синий (логика) и оранжевый (LLM). Иконки Lucide: `Layers`, `Settings`.
**Ключевые тезисы:**
* Сильная модель не означает надёжного исполнения. Agent = Model + Harness (Обвязка).
* **Аналогия ОС:** Модель выступает в роли CPU, контекстное окно — RAM, Harness — операционная система, управляющая ресурсами.
* **Пять подсистем Harness:** 1. Инструкции (файл `` как единый источник истины), 2. AI-агент, 3. Сохранение прогресса, 4. Инструменты/Среда, 5. Рантайм верификация.
* Задача Harness — ограничить поведение агента и не давать ему преждевременно объявлять задачу выполненной без сквозного тестирования.
