# Сценарий видео: Ландшафт AI и основы веб-технологий

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Слайд 1. Крупный план спикера. Появляется анимированная схема Клиент-Сервер (REST API).** | Добро пожаловать на первую неделю. Любая ИИ-автоматизация начинается с умения общаться с внешним миром. REST API — это мост между системами. Для извлечения данных мы используем метод GET, для создания — POST, для обновления — PUT, а для удаления — DELETE. Заголовки, или Headers, передают скрытые метаданные, а параметры Query и Body — саму суть нашего запроса. |
| **Слайд 2. Инфографика "Светофор": Зеленый (2xx), Желтый (3xx), Красный (4xx/5xx).** | Отправив запрос, мы всегда получаем статус-код. 200 означает успех, 401 — ошибку авторизации, а 500 — падение на стороне сервера. Особое внимание уделяйте ошибке 429 — превышению лимитов (Rate Limiting). В скриптах мы обязаны читать заголовок Retry-After и использовать Exponential Backoff, чтобы не сжечь бюджет на бесконечных повторах. |
| **Слайд 3. Интерфейс редактора кода с JSON. Подсветка синтаксиса, стрелки указывают на {} и [].** | Чтобы системы понимали друг друга, мы используем JSON — формат, состоящий из пар "ключ-значение" в фигурных скобках и списков в квадратных. JSON — это "родной язык" для LLM. Чтобы ИИ не галлюцинировал форматы, мы используем JSON Schema. Это заставляет модель выдавать строгий, предсказуемый результат, пригодный для конвейеров автоматизации. |
| **Слайд 4. Анимация события: Внешняя система (CRM) мгновенно пушит данные (POST) в узел Webhook.** | Вместо того чтобы постоянно спрашивать сервер "есть ли новые данные", мы используем вебхуки. Это инверсия контроля: система сама отправляет нам POST-запрос при наступлении события. Главное правило здесь — избежать ошибки "Timeout Drop". Ваш слушатель обязан мгновенно вернуть серверу статус 200 OK, и только потом передавать тяжелую полезную нагрузку агенту на обработку. |
| **Слайд 5. Схема DAG (Направленный ациклический граф) с иконками GitHub, OpenWeatherMap и LLM.** | Перейдем к сквозной практике. В паттерне "Оркестратор-Воркер" мы жестко собираем данные из публичных API, например GitHub и Weather, объединяем их и передаем языковой модели. Модель выступает только как синтезатор, чтобы написать итоговый markdown-брифинг. Это исключает риск того, что автономный агент начнет фантазировать данные. |
| **Слайд 6. Визуализация сейфа (Credentials). Ключи (Bearer) и схема делегированного доступа (OAuth2).** | Доступ к корпоративным данным требует жесткой защиты. Поскольку HTTP-протокол не имеет памяти, каждый запрос требует авторизации. Мы используем Bearer-токены для серверной коммуникации и протокол OAuth2 для действий от лица пользователя. Золотое правило безопасности: ключи никогда не должны попадать в контекстное окно LLM, они хранятся в защищенных хранилищах или.env файлах. |
| **Слайд 7. Окно терминала с выполнением команды `uv venv` и кусок Python кода со словарями.** | Если вы идете по пути разработчика, вам нужен Python. Для изоляции сред и предотвращения конфликтов зависимостей мы используем сверхбыстрый менеджер `uv` вместо глобальных библиотек. Синтаксис Python, особенно понимание списков и словарей, абсолютно необходим, так как они являются прямыми эквивалентами массивов и объектов JSON. |
| **Слайд 8. Циклическая диаграмма ReAct: Observe (наблюдение) -> Think (мысль) -> Act (действие).** | Голая языковая модель — это не агент. Augmented LLM, или дополненная модель, обретает автономию благодаря инструментам, памяти и базовому циклу рассуждений. В цикле ReAct модель анализирует контекст, размышляет над задачей и генерирует вызов инструмента. Этот цикл `while` крутится до тех пор, пока задача не будет решена. |
| **Слайд 9. Сравнительная таблица или сплит-экран: Прямая линия (Workflow) против Замкнутого цикла (Agent).** | Важнейший навык архитектора — знать, когда агент не нужен. Линейные процессы (Workflows) контролируются инженером: они детерминированы, дешевы и легко отлаживаются. Автономные агенты сами определяют поток управления. Они нужны для сложных исследовательских задач, но потребляют примерно в 15 раз больше токенов и требуют строгого контроля. |
| **Слайд 10. Архитектурный чертеж: LLM = CPU, ОЗУ = Контекст, Harness = ОС. Защитные барьеры и тесты.** | Сильная модель не гарантирует надежного исполнения. Сама по себе LLM — это лишь процессор. Ваш тестовый стенд, или Agent Harness — это операционная система. Harness управляет инструкциями, состоянием, диспетчеризацией инструментов и, самое главное, не дает агенту преждевременно объявить задачу выполненной без прохождения сквозных интеграционных тестов [23-25]. |
