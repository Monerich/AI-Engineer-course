# Сценарий видео: Голосовые и Мультимодальные Агенты

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Scene 1:** <br>Средний план спикера. Фон сменяется на неоново-синюю топологию сети, контрастно показывающую разницу между REST API и постоянными двунаправленными WebSocket-соединениями. | Добро пожаловать на 23-ю неделю курса Архитекторов ИИ-Автоматизации. Сегодня мы оставляем позади безопасный мир REST API без сохранения состояния и погружаемся в динамичную физику инфраструктуры медиа-стриминга. Чтобы создать по-настоящему разговорный ИИ, вы должны научиться управлять непрерывными бинарными потоками данных, передаваемыми через WebSockets, и понимать компромиссы между TCP и UDP для работы с медиа в реальном времени. |
| **Scene 2:** <br>Разделенный экран. Слева — классическое корпоративное IVR-дерево. Справа — SIP-транк Twilio, маршрутизирующий аудиосигнал 8kHz напрямую в облачную среду Python. | Мы начнем техническую реализацию с интеграции ИИ-телефонии. Вы узнаете, как настроить SIP-транки Twilio для перехвата входящих звонков и настройки вебхуков для исходящего обзвона. Мы разберем специфику обработки узкополосного аудио с частотой 8 килогерц, чтобы ваш ИИ мог нативно распознавать стандартные телефонные сигналы без потери производительности. |
| **Scene 3:** <br>Классическое телефонное дерево рассыпается, превращаясь в граф намерений (intent-graph). Загорается узел «Triage Agent», направляющий устный запрос клиента к специализированным суб-агентам. | Эта телефонная инфраструктура позволит нам строить динамические голосовые сценарии. Мы убиваем традиционные меню «Нажмите 1 для связи с отделом продаж». Вместо этого вы научите LLM выступать в роли агентов первичной сортировки (triage), которые оценивают запрос клиента в реальном времени, делают мгновенные запросы в CRM и динамически перенаправляют вызов на основе устного намерения. |
| **Scene 4:** <br>Спикер уходит на второй план, на экране — высококонтрастная схема WebRTC (mesh-сеть) с выделенными узлами Selective Forwarding Units (SFU) для ультранизкой задержки. | Для сценариев, требующих ультранизкой задержки, мы развернем комнаты LiveKit. Вы настроите Selective Forwarding Units (SFU) для эффективной маршрутизации аудио- и видеопотоков. Используя протокол WebRTC поверх UDP, мы полностью обходим узкие места TCP-пакетов, создавая бесшовную peer-to-peer архитектуру с задержкой менее 300 миллисекунд. |
| **Scene 5:** <br>Тёмная тема IDE за плечом спикера. По вертикали бежит быстрый поток строк base64, подсвеченный тегами кодировки `pcm16`. | Чтобы связать эти комнаты LiveKit с нашим ИИ, необходимо спроектировать кастомные аудиобуферы на WebSockets. Вы напишете логику на Python для захвата непрерывного аналогового звука, нарезки его на стандартизированные чанки `pcm16` с частотой 24 килогерца и кодирования в base64 для полнодуплексной передачи. |
| **Scene 6:** <br>Появляется блок-схема классического «Каскадного пайплайна» (STT → LLM → TTS). Мигает красная метка: «Накопление задержки» (Latency Accumulation). | Исторически разработчики полагались на каскадные пайплайны, склеивая облачные движки STT и TTS. Мы соберем один такой, чтобы вы поняли механику, но вы сразу увидите его фатальный недостаток: накопление задержки. Каскадные системы вводят паузы в 2-3 секунды и навсегда уничтожают паралингвистические данные: эмоции, ритм и тон. |
| **Scene 7:** <br>Крупный план Python-скрипта с реализацией `VoicePipelineAgent`. Текст подсвечивает фреймворк D.O.E. (Directive, Orchestration, Execution). | Для решения этой проблемы оркестрации мы используем LiveKit Agent Python SDK. Думайте об этом рантайме на Python как об операционной системе для вашего агента. Мы будем строго применять фреймворк DOE — разделяя Директивы, Оркестрацию и Исполнение — чтобы гарантировать детерминированную надежность инструментов даже в хаосе живого звонка. |
| **Scene 8:** <br>Экран вспыхивает яркой схемой, показывающей прямую петлю бесконечности от пользователя к модели на базе OpenAI Realtime API. Таймер показывает «< 100ms TTFB». | И это подводит нас к самому острию индустрии: голосовым циклам с суб-100мс задержкой через OpenAI Realtime API. Используя нативные Omni-модели, обрабатывающие сырое аудио напрямую, мы полностью избавляемся от текстовых посредников. Вы научитесь инициализировать строгие границы сессий и передавать схемы инструментов через WebSockets для достижения человеческой скорости реакции. |
| **Scene 9:** <br>Абстрактная визуализация Jitter-буфера, который расширяется и сжимается, поглощая лаги сети. Рядом блок памяти сжимает старые текстовые логи. | Но потоковая передача звука влечет за собой сетевую энтропию. Мы внедрим критические оптимизации задержек WebSockets, написав асинхронные Jitter-буферы для маскировки потери пакетов. Более того, чтобы ваш агент не упал из-за переполнения контекста во время 45-минутного звонка, мы создадим механизмы динамической компакции токенов. |
| **Scene 10:** <br>Мигает красная тревога «Barge-In» (Прерывание). Блок-схема показывает разделение потока выполнения: безопасные задачи отменяются, а запись в базу данных защищена. | Наконец, человеческое общение хаотично. Люди перебивают. Мы задействуем серверный Voice Activity Detection, чтобы улавливать эти прерывания в реальном времени. Вы освоите паттерны динамической отмены — мгновенную очистку аудитоочередей и хирургическую остановку задач в Python, защищая при этом необратимые записи в БД через `asyncio.shield` от повреждения состояния. Давайте начнем. |
