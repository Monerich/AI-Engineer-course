# Сценарий видео: Векторные базы данных и RAG в автоматизациях

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Сцена 1:** Спикер в студии. 3D-анимация: слова превращаются в числа и группируются в многомерном пространстве (векторы). | Добро пожаловать на седьмую неделю. Сегодня мы разбираем семантический поиск. В отличие от традиционного поиска по точным совпадениям, мы преобразуем текст в числа — многомерные векторы, или эмбеддинги. Это позволяет находить смысловую близость концепций путем математического вычисления косинусного сходства между ними. |
| **Сцена 2:** Логотипы баз данных: Supabase, Pinecone, Qdrant. Схема облачного и локального хранения. | Для хранения эмбеддингов нужны специализированные векторные базы данных. Мы разберем Qdrant для локальных задач, Pinecone для облачных serverless-индексов и Supabase, который благодаря расширению pgvector позволяет хранить векторы рядом с обычными таблицами в PostgreSQL. |
| **Сцена 3:** Скринкаст n8n. Узел `Recursive Character Text Splitter` разрезает документ, данные идут в ноду OpenAI Embeddings, затем в базу. | В n8n нельзя просто загрузить 50-страничный PDF в нейросеть. Мы строим пайплайн: разбиваем документ на смысловые фрагменты через Recursive Character Text Splitter, прогоняем их через модель эмбеддингов и загружаем (Upsert) полученные векторы в базу. |
| **Сцена 4:** Настройки ноды `Vector Store Retriever` в n8n. Курсор устанавливает лимит score > 0.4. | Для извлечения знаний мы используем ноду Vector Store Retriever. Когда пользователь задает вопрос, инструмент векторизует его и ищет «ближайших соседей». Критически важно установить порог релевантности (например, score > 0.4), чтобы отсекать информационный мусор до того, как он попадет к агенту. |
| **Сцена 5:** Окно чата. Агент безупречно отвечает на сложный вопрос по регламенту, опираясь на внутренний PDF. | Переходим к практике. Мы собираем RAG-агента — корпоративного чат-бота для справки по продукту. Мы задаем жесткий системный промпт: «Отвечай только на основе предоставленного релевантного контекста. Ничего не придумывай». Это полностью исключает галлюцинации. |
| **Сцена 6:** Список из 20 результатов поиска динамически пересортировывается. Лучшие ответы поднимаются наверх. | Векторный поиск быстр, но иногда выдает приблизительные результаты. Для максимальной точности мы интегрируем API Cohere Rerank. Реранкер выступает в роли «судьи», который переоценивает сырую выдачу и выталкивает самые релевантные куски текста на самый верх контекстного окна. |
| **Сцена 7:** IDE с темной темой. Пишется Python-код с использованием метода `openai.embeddings.create`. | Визуальные узлы удобны, но для обработки сотен тысяч корпоративных документов мы переходим на Python. Программное создание эмбеддингов через `openai.embeddings.create` позволяет гибко управлять батчами, очищать данные и обходить лимиты API при масштабной загрузке. |
| **Сцена 8:** Дашборд LangSmith с зелёными галочками напротив метрик Context Relevance и Groundedness. | Как доказать бизнесу, что ваш RAG надежен? Нужна инженерия оценок. Мы внедряем автоматизированные тесты (RAG Evals), где LLM-судья замеряет полноту извлеченного контекста (Recall) и отсутствие выдуманных фактов, гарантируя качество до релиза. |
| **Сцена 9:** Огромный блок текста. Внимание ИИ фокусируется только на начале и конце, игнорируя середину. | Если загрузить в промпт слишком много найденных фрагментов, LLM забудет критическую информацию — это эффект «Lost in the Middle». Мы настроим обвязку так, чтобы минимизировать этот эффект через усечение и жесткую компакцию контекста. |
| **Сцена 10:** Схема слияния двух потоков: Dense Vectors (векторы) и Sparse BM25 (ключевые слова). | Чистые векторы плохо ищут точные артикулы или коды ошибок. Для production-систем мы проектируем гибридный поиск. Сочетая семантическое понимание плотных векторов с лексической точностью разреженного алгоритма BM25, мы получаем безупречный контекст для нашего агента. |
