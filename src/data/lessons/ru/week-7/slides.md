# Презентация: Векторные базы данных и RAG в автоматизациях

📊 Слайд 1. Блок 1: Семантический поиск — векторные представления (embeddings) и косинусное сходство.
**Визуальный концепт:** Тёмная тема (HSL: 220, 15%, 15%). Иконки Lucide: `Network`, `Brain`. 3D-график, где семантически похожие слова сгруппированы вместе.
**Ключевые тезисы:**
* Традиционный поиск ищет точные совпадения, тогда как семантический поиск опирается на смысл текста, преобразуя его в многомерные числовые векторы (эмбеддинги),.
* Модели машинного обучения (например, Word2Vec) группируют семантически похожие концепции близко друг к другу в векторном пространстве,.
* Поиск релевантной информации осуществляется путем математического вычисления расстояния (косинусного сходства) между вектором запроса пользователя и векторами базы данных.

📊 Слайд 2. Блок 2: Развертывание векторных баз — настройка pgvector в Supabase, использование Pinecone и Qdrant.
**Визуальный концепт:** Светлая тема (HSL: 210, 20%, 98%). Иконки Lucide: `Database`, `Server`. Сравнение NoSQL базы с реляционной БД.
**Ключевые тезисы:**
* Векторные базы данных (ВБД) — это специализированные хранилища, созданные для индексирования и поиска многомерных векторов.
* Qdrant отлично подходит для локального развертывания, а Pinecone предоставляет облачные serverless-индексы с изоляцией по пространствам имен (namespaces),,.
* Supabase позволяет использовать расширение `pgvector` в реляционной базе PostgreSQL, что дает возможность хранить векторы рядом с классическими табличными данными,.

📊 Слайд 3. Блок 3: Векторизация в n8n — сквозной пайплайн разбивки, эмбеддинга и загрузки документов в векторную БД.
**Визуальный концепт:** Тёмная тема (HSL: 200, 50%, 15%). Иконки Lucide: `FileText`, `Scissors`. Анимация нарезки документа на фрагменты.
**Ключевые тезисы:**
* Крупные документы (например, PDF) нельзя загрузить целиком. Их необходимо разбить на смысловые фрагменты с помощью инструмента Recursive Character Text Splitter, сохраняя контекст.
* Фрагменты пропускаются через модель эмбеддингов (например, `text-embedding-3-small` от OpenAI), которая конвертирует текст в числа,.
* Полученные векторы вместе с метаданными загружаются (Upsert) в векторную базу данных для последующего использования ИИ-агентами,.

📊 Слайд 4. Блок 4: Retriever в n8n — настройка ноды Vector Store Retriever для поиска контекста по запросу.
**Визуальный концепт:** Светлая тема (HSL: 0, 0%, 100%). Иконки Lucide: `Search`, `DatabaseZap`. Интерфейс настройки порога релевантности в n8n.
**Ключевые тезисы:**
* Когда пользователь задает вопрос, его текст векторизуется той же моделью, что использовалась при загрузке данных, чтобы найти "ближайших соседей",.
* В n8n нода Vector Store Retriever действует как инструмент поиска, извлекая релевантные фрагменты контекста.
* Архитектор настраивает порог релевантности (например, score > 0.4), чтобы ИИ-агент игнорировал мусорные совпадения и работал только с точными данными.

📊 Слайд 5. Блок 5: Практикум: Справка по продукту — сборка чат-агента RAG, отвечающего по регламенту компании.
**Визуальный концепт:** Сплит-скрин. Иконки Lucide: `MessageSquare`, `ShieldCheck`. Бот поддержки отвечает на запрос из базы знаний.
**Ключевые тезисы:**
* RAG (Retrieval-Augmented Generation) позволяет обогащать ответы языковой модели частными знаниями компании.
* Агент поддержки настраивается со строгим системным промптом, предписывающим отвечать только на основе найденного контекста,.
* Интеграция RAG резко снижает галлюцинации LLM и делает ИИ надежным экспертом по внутренним базам данных (FAQ, политики, инструкции),.

---

📊 Слайд 6. Блок 6: Cohere Rerank — интеграция API Cohere Rerank для фильтрации результатов векторного поиска.
**Визуальный концепт:** Тёмная тема (HSL: 220, 10%, 10%). Иконки Lucide: `ListOrdered`, `TrendingUp`. Пересортировка списка результатов.
**Ключевые тезисы:**
* Векторный поиск работает очень быстро, но он приблизителен и часто выдает много контекстуального "шума",.
* Для повышения точности применяется двухэтапное извлечение. Реранкер (например, модель Cohere) выступает в роли вторичного "судьи",.
* Реранкер переоценивает найденные результаты и выталкивает самые релевантные куски текста на верхние позиции контекстного окна.

📊 Слайд 7. Блок 7: Программное создание векторных представлений текстов на чистом Python с использованием openai.embeddings.
**Визуальный концепт:** Тема редактора кода (Тёмная). Иконки Lucide: `FileCode2`, `Cpu`.
**Ключевые тезисы:**
* Для масштабных задач загрузки корпоративных знаний визуальные редакторы могут быть медленными; эффективнее использовать Python.
* Использование API `openai.embeddings.create` позволяет программно конвертировать текстовые массивы в векторы размерности 1536,.
* При реализации на Python разработчик может управлять скоростью отправки батчей, настраивать очистку текстов и обрабатывать ошибки.

📊 Слайд 8. Блок 8: Оценка качества RAG: ключевые метрики извлечения релевантного контекста.
**Визуальный концепт:** Светлая тема (HSL: 210, 20%, 98%). Иконки Lucide: `BarChart`, `Target`. Дашборд с метриками (LangSmith).
**Ключевые тезисы:**
* Невозможно улучшать RAG-систему без метрик. Инженерия обвязок требует создания автоматизированных систем оценки (evals).
* Оценки проверяют: нашел ли Retriever правильные документы (Recall) и сгенерировал ли агент ответ без галлюцинаций на основе этого контекста.
* Использование платформ (таких как LangSmith или Inspect) позволяет блокировать деградацию RAG-пайплайна до деплоя в продакшен.

---

📊 Слайд 9. Блок 9: Минимизация эффекта «Lost in the Middle» (потеря инфы в середине контекста).
**Визуальный концепт:** Тёмная тема (HSL: 0, 0%, 15%). Иконки Lucide: `Minimize2`, `AlertTriangle`.
**Ключевые тезисы:**
* Исследования доказали, что при загрузке множества чанков LLM хорошо помнит начало и конец промпта, но забывает информацию посередине (Lost in the Middle).
* Чем больше извлеченных документов попадает в промпт (Instruction Bloat), тем хуже модель решает задачу.
* Решением является усечение лишних данных, компакция истории и вынос нерелевантной информации за пределы основного контекста,.

---

📊 Слайд 10. Блок 10: Проектирование гибридного поиска (сочетание плотных векторов и разреженного BM25).
**Визуальный концепт:** Светлая тема (HSL: 0, 0%, 100%). Иконки Lucide: `Layers`, `SearchCheck`. Слияние двух параллельных потоков.
**Ключевые тезисы:**
* Векторный поиск улавливает смысл, но часто не справляется с точными запросами (например, артикулы товаров или точные ID ошибок),.
* Классический разреженный поиск (BM25) идеально находит точные совпадения ключевых слов,.
* Гибридный поиск запускает оба метода параллельно и математически сливает результаты алгоритмом Reciprocal Rank Fusion, гарантируя превосходную лексическую и семантическую точность,.
