# Сценарий видео: Наблюдаемость и Трейсинг в LangSmith

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Сцена 1: Крупный план спикера, затем переход к схеме сквозной трассировки архитектуры OpenTelemetry (иконка Activity).** | Добро пожаловать на 18-ю неделю: Наблюдаемость и Трейсинг. OpenTelemetry предоставляет стандартизированный фреймворк для трассировки ИИ-приложений. Он обеспечивает видимость каждого шага, позволяя отслеживать вызовы LLM, выполнение инструментов и логику маршрутизации. Интеграция систем трассировки жизненно важна для выявления инфраструктурных ошибок до того, как они дойдут до пользователя. |
| **Сцена 2: Визуальный процесс в n8n, показывающий передачу данных во внешнюю базу данных (иконка DatabaseExport).** | При работе с n8n вам необходимо продвинутое логирование. Данные о каждом выполнении воркфлоу должны быть экспортированы во внешние хранилища и системы аналитики. Настройка такого экспорта критически важна для аудита ошибок и анализа исторических сбоев в продакшене, когда клиент пишет вам посреди ночи. |
| **Сцена 3: Тёмный интерфейс с иерархическим графом (DAG) внутри LangSmith или Langfuse (иконка Eye).** | Платформы вроде LangSmith и Langfuse обеспечивают нативный трейсинг, делая каждый шаг рассуждений агента видимым. Вы можете просматривать точные промпты, ответы модели и использование инструментов в каждом узле. Это позволяет проводить точечную отладку и внедрять оценки (evals) качества работы. |
| **Сцена 4: Столбчатая диаграмма, показывающая подсчет токенов и затраты на API (иконка DollarSign).** | Мульти-агентные системы могут потреблять до 15 раз больше токенов, требуя строгой финансовой дисциплины. Финансовый мониторинг должен отслеживать метрику cost-per-task и общие затраты на вызовы API. Детальный трейсинг позволяет увидеть, где именно кэширование промптов может радикально сэкономить ваш бюджет. |
| **Сцена 5: Линейный график времени отклика (latency), на котором выделена задержка до первого токена (иконка Timer).** | Скорость работы критична для пользовательского опыта. Мониторинг метрик вроде p95 latency помогает выявить узкие места. Трассировка позволяет отличить время генерации первого токена (TTFT) самой моделью от инфраструктурных задержек при параллельных вызовах инструментов. |
| **Сцена 6: Всплывающие уведомления в интерфейсах Slack и Telegram, сигнализирующие о достижении лимитов (иконка BellRing).** | Системы мониторинга обязаны включать алерты на превышение стоимости, рост доли отказов и деградацию задержек. Настройте автоматическую маршрутизацию уведомлений об ошибках в Telegram или Slack для немедленного реагирования. Это предотвратит ситуации, когда зациклившийся агент незаметно сжигает ваш баланс. |
| **Сцена 7: Окно IDE с Python-кодом, где выделены декораторы SDK LangSmith/Langfuse (иконка Code).** | В разработке на Python SDK платформ наблюдаемости интегрируются через декораторы функций, оборачивая логику в иерархические спаны. Это критически важно для трассировки "холодных стартов" — фазы инициализации контекста и сборки промпта еще до того, как происходит фактический вызов LLM. |
| **Сцена 8: Анимация пятислойного защитного щита (Harness Engineering, Лекция 1) (иконка ShieldAlert).** | Переходим к Harness Engineering. Лекция 1 учит нас: сильная модель не гарантирует надёжного исполнения. Та же самая модель дает принципиально разные результаты в зависимости от обвязки. Главный принцип: когда что-то ломается, не меняйте сразу модель — сначала проверьте и почините ваш harness. |
| **Сцена 9: Схема изолированного Docker-контейнера (песочницы) с потоком обратной связи (иконка Box).** | Лекция 2 объясняет, что harness — это замкнутая рабочая система. Она состоит из изолированной среды исполнения (песочницы), доступных инструментов и системы обратной связи. Подсистема обратной связи перехватывает логи упавших тестов (симуляция ввода) и автоматически возвращает их агенту для самоисправления. |
| **Сцена 10: Древовидная структура директорий с выделенным файлом и локальными инструкциями (иконка FolderTree).** | Наконец, Лекции 3 и 4. Ваш код-репозиторий должен стать единственным источником правды. Огромные файлы промптов перегружают контекст и вызывают эффект "Lost in the Middle", когда модель игнорирует правила в середине текста. Инструкции необходимо разделять по файлам и подгружать динамически, используя принцип прогрессивного раскрытия. |
