# Презентация: Безопасность, Инъекции и Песочницы

📊 Слайд 1. Блок 1: Концептуальная безопасность LLM
**Visual Layout Concept:** Темная тема HSL (`hsl(222, 47%, 11%)`). Иконки Lucide: `ShieldAlert`, `BrainCircuit`. 
* Уязвимости AI-агентов концептуально делятся на внутренние (взломы самого LLM-«мозга») и внешние (угрозы от памяти, среды и взаимодействия с другими агентами).
* Векторы атак включают джейлбрейки, инъекции промптов и отравление данных, искажающие процесс принятия решений.

---
📊 Слайд 2. Блок 2: Санитаризация вебхуков
**Visual Layout Concept:** Светлая тема HSL (`hsl(210, 40%, 98%)`). Иконки Lucide: `Filter`, `Webhook`. 
* Все входящие данные из вебхуков должны проходить жесткую санитаризацию перед отправкой в контекст модели, так как нельзя доверять внешнему вводу.
* Непрямые инъекции промптов (Indirect Prompt Injection) используют внешние ресурсы (веб-страницы, email) для внедрения вредоносных команд.

---
📊 Слайд 3. Блок 3: Защита от System Prompt Leakage
**Visual Layout Concept:** Темная тема HSL (`hsl(222, 47%, 11%)`). Иконки Lucide: `FileKey`, `Lock`. 
* System Prompt Stealing — атака, нацеленная на извлечение системных инструкций агента для раскрытия его архитектуры и уязвимостей.
* Защита строится на использовании классификаторов безопасности и фильтрации, которые предотвращают утечку внутренних правил модели.

---
📊 Слайд 4. Блок 4: OWASP Top 10 для LLM
**Visual Layout Concept:** Светлая тема HSL (`hsl(0, 0%, 100%)`). Иконки Lucide: `ListOrdered`, `ShieldCheck`. 
* Стандарт OWASP Top 10 для LLM систематизирует критические риски перед выходом в продакшен.
* Главные угрозы включают инъекции, утечку конфиденциальных данных и «Excessive Agency» (выдачу агенту избыточных полномочий).

---
📊 Слайд 5. Блок 5: Настройка Guardrails
**Visual Layout Concept:** Темная тема HSL (`hsl(222, 47%, 11%)`). Иконки Lucide: `Fence`, `CheckCircle`. 
* Guardrails (защитные барьеры) валидируют вывод LLM, гарантируя его синтаксическую корректность, безопасность и отсутствие токсичности.
* Входные фильтры блокируют джейлбрейки, а выходные — предотвращают утечку PII (персональных данных) и генерацию опасного контента.

---
📊 Слайд 6. Блок 6: Разграничение прав к БД
**Visual Layout Concept:** Светлая тема HSL (`hsl(210, 40%, 98%)`). Иконки Lucide: `Database`, `Key`. 
* Интеграция БД требует строгого соблюдения принципа наименьших привилегий (Principle of Least Privilege).
* Используйте Role-Based Access Control (RBAC) и выдавайте агенту только read-only права, если изменение данных не требуется задачей.

---
📊 Слайд 7. Блок 7: Безопасный запуск кода (E2B / Daytona)
**Visual Layout Concept:** Темная тема HSL (`hsl(222, 47%, 11%)`). Иконки Lucide: `Box`, `Terminal`. 
* Никогда не выполняйте сгенерированный LLM код напрямую на основном хосте.
* Все выполнение кода должно происходить в эфемерных, изолированных песочницах (sandboxes), таких как Modal, E2B или Daytona.

---
📊 Слайд 8. Блок 8: Взаимодействие агентов с ОС хоста
**Visual Layout Concept:** Светлая тема HSL (`hsl(0, 0%, 100%)`). Иконки Lucide: `Server`, `Cpu`. 
* Безопасная архитектура разделяет контекст модели («мозг») и среду выполнения («руки»).
* Учетные данные и API-ключи брокерятся вне контекста LLM, чтобы инъекция не могла привести к краже секретов.

---
📊 Слайд 9. Блок 9: Ограничение сетевых политик внутри песочниц
**Visual Layout Concept:** Темная тема HSL (`hsl(222, 47%, 11%)`). Иконки Lucide: `Network`, `WifiOff`. 
* Сетевой доступ изнутри песочницы должен быть по умолчанию запрещен или строго ограничен.
* Это блокирует атаки, при которых скомпрометированный агент пытается переслать (эксфильтровать) данные на внешние серверы злоумышленника.

---
📊 Слайд 10. Блок 10: Подтверждение человеком (HITL)
**Visual Layout Concept:** Светлая тема HSL (`hsl(210, 40%, 98%)`). Иконки Lucide: `UserCheck`, `AlertOctagon`. 
* Автономным моделям запрещено доверять необратимые действия (финансы, удаление данных, массовые рассылки) без контроля.
* Внедряйте шаги Human-In-The-Loop (HITL) через прерывания, ожидая явного подтверждения оператором (например, в Slack) перед выполнением критического инструмента.
