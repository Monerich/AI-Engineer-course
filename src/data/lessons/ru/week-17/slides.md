# Презентация: Продвинутый RAG для продакшна

📊 Слайд 1. Блок 1 (ИИ-Инженер / Автоматизация): Умный чанкинг — продвинутые стратегии нарезки текстов на чанки с умным перекрытием.
* **Визуальный концепт:** Тёмная тема (HSL: 220, 10%, 15%). Иконка Lucide: `scissors`. Схема, показывающая сплошной текст документа, разбивающийся на перекрывающиеся кусочки пазла.
* **Ключевые пункты:**
 * Продвинутые семантические чанкеры обрабатывают исходные документы, сохраняя смысловой контекст с помощью иерархии заголовков.
 * Умное перекрытие (overlap) гарантирует, что критически важный контекст не будет разорван на середине концепции или предложения.

---

📊 Слайд 2. Блок 2 (ИИ-Инженер / Автоматизация): Parent-Document Retriever — сохранение широкого контекста при точечном извлечении чанков.
* **Визуальный концепт:** Светлая тема (HSL: 210, 20%, 95%). Иконка Lucide: `file-search`. Схема: выделенный маленький блок текста со стрелками, расширяющимися до родительского документа.
* **Ключевые пункты:**
 * Улучшает RAG за счет поиска по компактным и точным векторам для максимального семантического соответствия.
 * После нахождения совпадения генеративной модели передается более широкий окружающий контекст (родительский документ) для более глубокого понимания логики.

---

📊 Слайд 3. Блок 3 (ИИ-Инженер / Автоматизация): Переранжирование — интеграция API Cohere Rerank для фильтрации результатов векторного поиска.
* **Визуальный концепт:** Тёмная тема (HSL: 280, 15%, 15%). Иконка Lucide: `sort-asc`. Диаграмма воронки, фильтрующей 100 первоначальных результатов в очищенный Топ-5.
* **Ключевые пункты:**
 * Стандартный векторный поиск работает невероятно быстро, но возвращает десятки приблизительных и "шумных" совпадений.
 * Ранжировщики (Rerankers) переоценивают эти результаты с помощью более сложной логики, чтобы гарантировать, что переданные в LLM чанки максимально релевантны.

---

📊 Слайд 4. Блок 4 (ИИ-Инженер / Автоматизация): Query Translation — генерация субзапросов и переписывание пользовательских запросов.
* **Визуальный концепт:** Светлая тема (HSL: 40, 30%, 95%). Иконка Lucide: `git-branch`. Диаграмма одного запроса пользователя, распадающегося на три параллельных субзапроса.
* **Ключевые пункты:**
 * Агенты используют контекстно-зависимое расширение запросов для генерации нескольких уточнений вместо одного прямого поиска.
 * Сложные запросы декомпозируются на мелкие логические шаги, позволяя агенту извлекать информацию последовательно.

---

📊 Слайд 5. Блок 5 (ИИ-Инженер / Автоматизация): Парсинг таблиц и схем — разбор сложных неструктурированных документов со встроенными таблицами.
* **Визуальный концепт:** Тёмная тема (HSL: 120, 10%, 15%). Иконка Lucide: `table`. Схема сложной PDF-таблицы, плавно конвертирующейся в структурированный JSON/Markdown.
* **Ключевые пункты:**
 * Наивный чанкинг разрушает логические связи в таблицах и заголовки столбцов.
 * Продвинутые парсеры макетов (Layout Parsers) автоматически обрабатывают сложную верстку, встроенные таблицы и изображения, сохраняя структурный смысл для векторной базы.

---

📊 Слайд 6. Блок 6 (ИИ-Инженер / Автоматизация): Векторное масштабирование — масштабирование индексов векторных баз данных под высокой нагрузкой.
* **Визуальный концепт:** Светлая тема (HSL: 200, 25%, 95%). Иконка Lucide: `database-zap`. Диаграмма кластеризации SCaNN для быстрого поиска приближенных ближайших соседей.
* **Ключевые пункты:**
 * Поиск по эмбеддингам всегда требует жесткого компромисса между скоростью вычислений и точностью.
 * Алгоритмы вроде SCaNN используются для кластеризации и оптимизации высоконагруженных запросов в базах данных миллиардного масштаба.

---

📊 Слайд 7. Блок 7 (Python-разработка): Реализация логики Self-RAG (Corrective RAG - CRAG) на Python для автооценки релевантности документов.
* **Визуальный концепт:** Тёмная тема (HSL: 0, 0%, 15%). Иконка Lucide: `refresh-ccw`. Схема цикла извлечения информации с гейтом самокоррекции.
* **Ключевые пункты:**
 * Внедряет валидацию и коррекцию, где агенты-оценщики перекрестно проверяют извлеченные знания для предотвращения галлюцинаций.
 * Позволяет пайплайну автономно отбрасывать нерелевантные документы и переписывать запрос перед попыткой генерации ответа.

---

📊 Слайд 8. Блок 8 (Создатель ИИ-Агентов / Агенты & Harness): Архитектура Оценщик-Оптимизатор (Evaluator-Optimizer) для контроля качества генерации.
* **Визуальный концепт:** Светлая тема (HSL: 260, 20%, 95%). Иконка Lucide: `check-circle`. Диаграмма: Агент-Генератор передает черновик изолированному Агенту-Критику.
* **Ключевые пункты:**
 * Мультиагентный цикл, где Генератор пишет черновик и передает его Оценщику, запущенному с абсолютно чистым контекстным окном.
 * Устраняет искажение невозвратных затрат, заставляя Оценщика строго проверять вывод по рубрике и возвращать конкретные "красные пометки" для доработки.

---

📊 Слайд 9. Блок 9 (Создатель ИИ-Агентов / Агенты & Harness): Проектирование сценариев 'заглушек' при отсутствии релевантных документов в векторной БД.
* **Визуальный концепт:** Тёмная тема (HSL: 350, 15%, 15%). Иконка Lucide: `shield-alert`. Диаграмма конечного автомата: Векторная БД -> `[]` -> маршрутизация на веб-поиск или человека.
* **Ключевые пункты:**
 * Реализация "Защитного UX", который жестко блокирует попытки LLM выдумать ответ при отсутствии внутренних знаний.
 * Активация адаптивного выбора источников для маршрутизации "голодных" запросов на внешние веб-API или эскалацию на человека (Human-in-the-Loop).

---

📊 Слайд 10. Блок 10 (Создатель ИИ-Агентов / Агенты & Harness): Расчет ключевых RAG-метрик: Recall (полнота), Precision (точность) и Hit Rate.
* **Визуальный концепт:** Светлая тема (HSL: 100, 20%, 95%). Иконка Lucide: `bar-chart`. Диаграмма CI/CD пайплайна, блокирующего регрессию на основе Золотого датасета.
* **Ключевые пункты:**
 * Заменяет интуитивное тестирование детерминированными оценками LLM-as-a-Judge для точного измерения производительности поиска.
 * Жестко отслеживает Hit Rate (найден ли нужный документ?), Contextual Recall (есть ли все факты?) и Contextual Precision (много ли мусора в чанках?).
