# Презентация: ИИ-ноды n8n и интеграция с LangChain

📊 Слайд 1. Блок 1 (ИИ-Инженер / Автоматизация): Архитектура ИИ-нод n8n — как под капотом собираются цепочки и агенты библиотеки LangChain.
* **Visual Layout Concept:** Темная тема `HSL(220, 15%, 16%)`. Древовидная схема, показывающая корневую ноду и подключенные суб-ноды. Иконка: Lucide `Cpu`.
* **Ключевые факты:**
 * Интеграция Advanced AI в n8n построена поверх популярной библиотеки LangChain.
 * Архитектура модульная: корневые узлы (например, AI Agent) подключаются к суб-узлам, таким как память, LLM-модели, парсеры и инструменты.
 * Такая структура позволяет визуально собирать сложные когнитивные пайплайны, разделяя логику обработки, хранилище и исполняемые инструменты.

📊 Слайд 2. Блок 2 (ИИ-Инженер / Автоматизация): Нода AI Agent в n8n — использование ИИ-агента в режиме Conversational Agent.
* **Visual Layout Concept:** Светлая тема `HSL(210, 20%, 98%)`. Схема: Пользователь -> AI Agent -> Провайдер LLM. Иконка: Lucide `Bot`.
* **Ключевые факты:**
 * Нода AI Agent может работать в нескольких режимах, включая Conversational Agent и ReAct Agent.
 * Агент автоматически обрабатывает входящие запросы, управляет диалогом и решает, когда нужно запросить внешние данные.
 * В системном промпте агента задаются его роль, ограничения и правила использования доступных инструментов.

📊 Слайд 3. Блок 3 (ИИ-Инженер / Автоматизация): Продвинутые инструменты (Tools) — разработка кастомных инструментов через HTTP-запросы и JS.
* **Visual Layout Concept:** Темная тема `HSL(222, 20%, 18%)`. Фрагмент кода на TypeScript с вызовом API. Иконка: Lucide `Wrench`.
* **Ключевые факты:**
 * Для нестандартных задач используются ноды Custom Code Tool или HTTP Request.
 * Разработка собственных комьюнити-нод для n8n требует описания логики в TypeScript-классе через метод `execute()`.
 * Внутри метода настраивается формирование JSON, авторизация через Credentials и отправка HTTP-запросов (например, через `this.helpers.httpRequest`).

📊 Слайд 4. Блок 4 (ИИ-Инженер / Автоматизация): Память чат-ботов — сохранение сессий диалогов: Window Buffer Memory vs внешняя БД pg/redis.
* **Visual Layout Concept:** Светлая тема `HSL(215, 25%, 95%)`. Сравнение временного буфера и базы данных PostgreSQL. Иконка: Lucide `Database`.
* **Ключевые факты:**
 * `Window Buffer Memory` сохраняет контекст только в рамках текущей сессии (в оперативной памяти), опираясь на динамический Session ID.
 * Для долгосрочных задач (long-horizon) необходимо использовать внешние хранилища через суб-ноды `Postgres Chat Memory` или `Redis Chat Memory`.
 * Сохранение контекста между сессиями решает проблему "амнезии" модели, позволяя восстановить диалог при перезапуске или сбое.

📊 Слайд 5. Блок 5 (ИИ-Инженер / Автоматизация): Импорт документов — загрузка документов (PDF, TXT, CSV) через Document Loaders для векторизации.
* **Visual Layout Concept:** Темная тема `HSL(210, 10%, 20%)`. Иконки различных форматов файлов, сливающиеся в единый поток данных. Иконка: Lucide `FileText`.
* **Ключевые факты:**
 * Для обработки файлов n8n использует бинарные данные, которые извлекаются через `Default Data Loader` и другие парсеры.
 * Документы трансформируются в унифицированный объект `Document`, содержащий текст (`page_content`) и метаданные.
 * Импорт структурированных (CSV) и неструктурированных (PDF) файлов является первым шагом перед их передачей в RAG-системы.

---

📊 Слайд 6. Блок 6 (ИИ-Инженер / Автоматизация): Стратегии разбивки текстов — работа с Text Splitters: заголовочный чанкинг, скользящее окно.
* **Visual Layout Concept:** Светлая тема `HSL(205, 30%, 96%)`. Ножницы, разрезающие текст на пересекающиеся блоки (chunks). Иконка: Lucide `Scissors`.
* **Ключевые факты:**
 * Загрузка целого документа в LLM приводит к "Instruction Bloat" и потере информации в середине текста (Lost in the Middle Effect).
 * `Recursive Character Text Splitter` и `Token Splitter` математически разбивают документы с заданным перекрытием (Overlap), сохраняя контекст на стыке фрагментов.
 * Для сохранения семантики Markdown-документов применяются кастомные скрипты заголовочного чанкинга.

📊 Слайд 7. Блок 7 (Python-разработка): Профессиональное использование официальных SDK OpenAI/Anthropic в Python • Управление таймаутами.
* **Visual Layout Concept:** Темная тема `HSL(225, 15%, 15%)`. Терминал с Python-скриптом, использующим `try/except`. Иконка: Lucide `TerminalSquare`.
* **Ключевые факты:**
 * Использование официальных SDK (например, Claude Agent SDK) позволяет разработчикам жестко контролировать циклы, таймауты и вызовы инструментов вне визуальной среды.
 * Python-скрипты позволяют внедрять хуки (PreToolUse, PostToolUse) и перехватывать ошибки исполнения.
 * Это снижает риск зависания системы из-за сетевых сбоев API.

📊 Слайд 8. Блок 8 (Создатель ИИ-Агентов / Агенты & Harness): Описание инструментов (JSON-схемы) для агента: промптинг модели на выбор нужного тула.
* **Visual Layout Concept:** Светлая тема `HSL(210, 15%, 98%)`. Строгая JSON-схема. Иконка: Lucide `FileJson`.
* **Ключевые факты:**
 * Названия инструментов и их JSON-схемы являются прямыми инструкциями для LLM.
 * Описание инструмента должно содержать 3–4 предложения: что он делает, когда его применять и какие у него ограничения.
 * Точное определение параметров (`enum`, типы данных) предотвращает галлюцинации и гарантирует вызов нужного API.

📊 Слайд 9. Блок 9 (Создатель ИИ-Агентов / Агенты & Harness): Защита от prompt injection через параметры и аргументы инструментов агента.
* **Visual Layout Concept:** Темная тема `HSL(0, 20%, 15%)`. Щит, защищающий блок кода. Иконка: Lucide `ShieldCheck`.
* **Ключевые факты:**
 * Разработка ИИ-интеграций требует санитайзинга пользовательского ввода и недоверия к выводу LLM при необратимых действиях (Human-in-the-loop).
 * Использование `PreToolUse`-хуков блокирует деструктивные команды (например, удаление файлов или сброс БД).
 * Системы аутентификации (Credentials) должны храниться в брокере секретов n8n, чтобы агенты никогда не видели сами ключи.

📊 Слайд 10. Блок 10 (Создатель ИИ-Агентов / Агенты & Harness): Трассировка рассуждений агента при вызове цепочек инструментов.
* **Visual Layout Concept:** Светлая тема `HSL(220, 10%, 96%)`. График спанов (spans) и задержек выполнения. Иконка: Lucide `Activity`.
* **Ключевые факты:**
 * Наблюдаемость (Observability) необходима для диагностики сложных мульти-агентных процессов, так как без нее отладка превращается в угадывание.
 * Платформы вроде LangSmith собирают OTEL-spans на каждый вызов модели, инструмента и подсчитывают латенси и токены.
 * Трассировка позволяет сформировать «диагностическую петлю», выявляя на каком именно этапе (проваленный промпт, ошибка API) прервалась работа агента.
