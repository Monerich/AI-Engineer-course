# Презентация: Оценки качества и регрессионный Harness

📊 Слайд 1. Блок 1: Датасеты из продакшна
* **Visual Layout Concept:** Dark mode (HSL: 220, 15%, 10%), акцентный синий неон.
* **Lucide Icon:** `Database`
* Золотые датасеты устаревают за месяцы, если их не обновлять.
* Эффективные тестовые датасеты должны расти из реальных падений в продакшне, а не из синтетических данных. 
* 1–10% stratified sampling на ошибках используется для сбора новых кейсов при высоких нагрузках.

---
📊 Слайд 2. Блок 2: Золотые датасеты (Golden Datasets)
* **Visual Layout Concept:** Light mode (HSL: 0, 0%, 98%), строгая типографика, минимализм.
* **Lucide Icon:** `Award`
* Golden Dataset должен состоять из 30–50 размеченных вручную вопросов разных уровней сложности.
* Эти эталонные выборки становятся основой для детерминированных грейдеров и LLM-as-a-judge в CI/CD.

---
📊 Слайд 3. Блок 3: Регрессионные тесты промптов
* **Visual Layout Concept:** Dark mode (HSL: 210, 20%, 15%), элементы интерфейса терминала.
* **Lucide Icon:** `GitMerge`
* Каждое обновление промпта или обвязки должно проходить через GitHub Actions с блокировкой PR при падении pass rate,.
* Обязательно проведение re-baseline оценок после каждого апгрейда LLM модели.

---
📊 Слайд 4. Блок 4: Мультимодельные бенчмарки
* **Visual Layout Concept:** Light mode (HSL: 210, 20%, 95%), графики сравнения.
* **Lucide Icon:** `Cpu`
* Оценка производительности разных моделей: Opus 4.7 показывает ~87.6% на SWE-bench Verified, а GPT-5.5 — 82.7% на Terminal-Bench 2.0.
* Бенчмарки диктуют маршрутизацию (routing): Haiku 4.5 для простых ходов, Opus 4.7 для планирования.

---
📊 Слайд 5. Блок 5: Оптимизация стоимости/качества
* **Visual Layout Concept:** Dark mode (HSL: 150, 10%, 15%), зеленые акценты.
* **Lucide Icon:** `DollarSign`
* Агрессивное кэширование промптов (Prompt Caching) позволяет экономить до 90% стоимости на системных промптах и инструментах.
* Использование Batch API дает скидку 50% для не-real-time нагрузки.

---
📊 Слайд 6. Блок 6: Дашборды регрессии
* **Visual Layout Concept:** Light mode (HSL: 0, 0%, 100%), дашборд-структура с виджетами.
* **Lucide Icon:** `LineChart`
* Интеграция с платформами observability (LangSmith, Braintrust, Inspect) для отслеживания дрейфа метрик.
* Команда `make eval` автоматизирует выдачу CI pass/fail summary и генерацию логов для дашборда.

---
📊 Слайд 7. Блок 7: Промышленное Е2Е-тестирование с Playwright
* **Visual Layout Concept:** Dark mode (HSL: 260, 20%, 10%), скриншоты UI-элементов.
* **Lucide Icon:** `MonitorPlay`
* Playwright MCP позволяет агенту-оценщику управлять браузером, кликать по элементам и валидировать визуальный результат работы (UI/UX),.
* Разделение ролей: один агент (Generator) пишет код, независимый агент (Evaluator) прогоняет Е2Е-тесты.

---
📊 Слайд 8. Блок 8: Harness Engineering. Лекция 7: Четкие границы задач
* **Visual Layout Concept:** Light mode (HSL: 40, 20%, 95%), канбан-схема.
* **Lucide Icon:** `ShieldAlert`
* Внедрение жесткого лимита WIP=1: в любой момент активна может быть только одна задача.
* Запрет на рефакторинг до верификации: ограничение приоритета предотвращает "overreach" и разрушение смежного кода,.

---
📊 Слайд 9. Блок 9: Harness Engineering. Лекция 8: Feature Lists
* **Visual Layout Concept:** Dark mode (HSL: 220, 10%, 12%), JSON-код на фоне.
* **Lucide Icon:** `ListChecks`
* Список фич — это конечный автомат (FSM) и фундаментальный примитив harness'а, а не текстовая памятка.
* Агент не может сам перевести задачу в статус `passing`; он обязан вызвать внешнюю команду верификации (экстернализация суждения),.

---
📊 Слайд 10. Блок 10: Harness Engineering. Лекции 9-10: Предотвращение преждевременного завершения
* **Visual Layout Concept:** Light mode (HSL: 0, 50%, 98%), красные текстовые маркеры.
* **Lucide Icon:** `Crosshair`
* LLM склонны к систематическому искажению уверенности (overconfidence), требуя независимых E2E тестов для закрытия Verification Gap.
* Harness должен выдавать «Красные пометки» (Red Ink Feedback) — не просто стектрейс, а прямую инструкцию, *как* архитектурно исправить ошибку,.
