# Презентация: Продвинутый n8n: Массивы данных и подпроцессы

📊 Слайд 1. Концепция Item Lists в n8n — параллельная обработка списков элементов, циклы и итерации.
**Визуальный концепт:** Светлая тема (HSL: 210, 20%, 98%). Иконки Lucide: `ListTree`, `Layers`. Визуализация JSON-массива, проходящего через ноду.
**Ключевые тезисы:**
* n8n обрабатывает данные в виде массивов JSON-объектов (Item Lists).
* По умолчанию каждая нода автоматически выполняется для каждого элемента в массиве (неявный цикл).
* Понимание работы массивов защищает от ошибок привязки элементов (Item linking) и переполнения памяти.

---

📊 Слайд 2. Итерация и циклы — разделение данных и порционная обработка через Split In Batches.
**Визуальный концепт:** Тёмная тема (HSL: 220, 15%, 15%). Иконки Lucide: `Repeat`, `Package`. Схема разделения потока данных на батчи.
**Ключевые тезисы:**
* Нода `Loop Over Items` (ранее Split In Batches) позволяет обрабатывать массив данных порциями (например, по 5 элементов).
* Узел имеет выходы `Loop` (для итераций) и `Done` (по завершении).
* Незаменим для обхода больших датасетов без риска "падения" системы от нехватки памяти (OOM).

---

📊 Слайд 3. Модульная архитектура n8n — вызов дочерних процессов через Execute Workflow.
**Визуальный концепт:** Светлая тема (HSL: 0, 0%, 100%). Иконки Lucide: `Workflow`, `Network`. Диаграмма родительского и дочернего процессов.
**Ключевые тезисы:**
* Узел `Execute Workflow` позволяет вызывать другие (дочерние) воркфлоу, превращая их в переиспользуемые микросервисы.
* Помогает избежать создания гигантских, нечитаемых схем из сотен нод.
* Упрощает отладку (Debugging Loops), так как сложная логика капсулируется в отдельном процессе.

---

📊 Слайд 4. Изоляция данных — передача и сохранение переменных состояния между родительским и дочерним флоу.
**Визуальный концепт:** Тёмная тема (HSL: 220, 10%, 10%). Иконки Lucide: `BoxSelect`, `Lock`. Схема передачи Payload.
**Ключевые тезисы:**
* Данные передаются из родительского воркфлоу в дочерний через триггер `Execute Workflow Trigger`.
* Дочерний процесс возвращает очищенный результат, реализуя принцип "Clean State Handoff" (Чистая передача).
* Изолирует переменные и предотвращает конфликт данных при параллельном выполнении.

---

📊 Слайд 5. Практикум: Новости с циклом — RSS-лента, фильтр 5 новостей, дочерний воркфлоу суммаризации.
**Визуальный концепт:** Сплит-скрин (светлый/тёмный). Иконки Lucide: `Rss`, `Bot`. Визуальный флоу из 4-5 нод.
**Ключевые тезисы:**
* Сборка пайплайна: нода `RSS Read` собирает статьи, нода `Limit` оставляет топ-5.
* Передача через `Loop` каждого URL в дочерний воркфлоу с AI-агентом для суммаризации текста.
* Возврат готовых выжимок в родительский флоу для агрегации и публикации.

📊 Слайд 6. Масштабирование n8n — настройка self-hosted n8n в режиме Redis Queue Mode для воркеров.
**Визуальный концепт:** Тёмная тема (HSL: 200, 50%, 15%). Иконки Lucide: `Server`, `Database`. Архитектура брокера сообщений.
**Ключевые тезисы:**
* Для Enterprise-нагрузок используется Queue Mode, где главный инстанс n8n выступает только маршрутизатором.
* Redis служит брокером сообщений, накапливая задачи.
* Task Runners (независимые воркеры) извлекают и параллельно обрабатывают задачи, обеспечивая горизонтальное масштабирование.

---

📊 Слайд 7. Python-разработка: Сложные трансформации массивов данных в Python внутри ноды Code в n8n.
**Визуальный концепт:** Тема редактора кода (Тёмная). Иконки Lucide: `Code2`, `Terminal`. Фрагмент Python-скрипта.
**Ключевые тезисы:**
* Узел `Code` позволяет использовать Python для сложных вычислений и трансформаций, недоступных базовым визуальным нодам.
* Доступ к входящим данным через объект `_input.all()`.
* Позволяет агрегировать, фильтровать массивы и использовать библиотеки (например, `json`, `math`) для глубокого парсинга.

---

📊 Слайд 8. Проектирование событийно-ориентированных (Event-Driven) архитектур автоматизации.
**Визуальный концепт:** Светлая тема (HSL: 210, 20%, 98%). Иконки Lucide: `Zap`, `Webhook`. Схема реакции на триггеры.
**Ключевые тезисы:**
* Системы реагируют на события (Webhooks, IMAP, изменения в БД), а не опрашивают их по таймеру (polling).
* Снижает нагрузку на серверы и обеспечивает обработку данных в реальном времени.
* Основа "тактического слоя автоматизации", отделяющая логику триггера от логики выполнения.

---

📊 Слайд 9. Управление очередями задач при работе с лимитами сторонних API (Rate Limiting).
**Визуальный концепт:** Тёмная тема с предупреждающими акцентами. Иконки Lucide: `ShieldAlert`, `Hourglass`.
**Ключевые тезисы:**
* Защита от ошибок HTTP 429 (Too Many Requests) с помощью настроек "Retry On Fail" и узла `Wait`.
* Паттерн "Экспоненциальная задержка" (Exponential Backoff) для безопасного повторения API-вызовов при сбоях.
* Батчинг (Batch API) и маршрутизация невалидных запросов в Dead Letter Queue (DLQ).

---

📊 Слайд 10. Распределенные состояния в low-code: защита данных при сбое на середине цикла.
**Визуальный концепт:** Светлая тема. Иконки Lucide: `DatabaseBackup`, `ShieldCheck`. Схема ACID-транзакции в БД.
**Ключевые тезисы:**
* Принцип Durable Execution (Надежное исполнение): сохранение прогресса в базу данных перед опасными шагами (API, LLM).
* Атомарные чекпоинты: статусы "pending", "processing", "completed".
* При падении контейнера или сервера процесс восстанавливает работу (Rewind & Fork) строго с прерванной точки без дублирования.
