# Сценарий видео: Продвинутый n8n: Массивы данных и подпроцессы

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Сцена 1:** Спикер в кадре. Рядом появляется графическое представление массива JSON. Иконки `ListTree` и `Layers`. | Добро пожаловать на пятую неделю. Сегодня мы погружаемся в продвинутую архитектуру n8n. Фундамент платформы — это Item Lists. n8n обрабатывает данные не как единичные строки, а как массивы JSON-объектов. По умолчанию каждая нода автоматически выполняется для каждого элемента в этом массиве. Это неявный цикл. Понимание того, как массивы проходят через узлы, защитит вас от ошибок привязки данных и переполнения памяти. |
| **Сцена 2:** Скринкаст интерфейса n8n. Узел `Loop Over Items` (ранее Split In Batches) делит поток данных на пакеты. | Но что делать, если на вход поступает 10 000 записей? Чтобы сервер не упал от нехватки памяти, мы используем явные циклы. Узел Loop Over Items, ранее известный как Split In Batches, позволяет разбить массив на порции — например, по 50 элементов за раз. Он имеет два выхода: Loop для итераций и Done по завершении. Это критически важный паттерн для обработки огромных датасетов. |
| **Сцена 3:** Архитектурная схема: Большой монолитный воркфлоу распадается на один родительский и несколько дочерних. Иконки `Workflow`. | По мере роста ваших автоматизаций холст превратится в нечитаемую паутину. Решение — модульная архитектура. С помощью узла Execute Workflow мы можем вызывать другие, дочерние процессы. Это превращает сложные куски логики в переиспользуемые микросервисы. Если вам нужно обновить логику работы с базой данных, вы меняете один дочерний флоу, и изменения применяются ко всем родительским процессам. |
| **Сцена 4:** Анимация передачи данных (Payload) от родительского процесса к узлу `Execute Workflow Trigger` с замком (изоляция). | Разделение на подпроцессы дает нам важнейшее свойство — изоляцию данных. Родительский процесс передает нужные переменные через узел Execute Workflow Trigger. Дочерний флоу отрабатывает в своей изолированной среде и возвращает только чистый результат. Это предотвращает конфликт глобальных переменных при параллельном выполнении и реализует принцип «чистой передачи состояния» в конце сессии. |
| **Сцена 5:** Сплит-скрин. Слева: нода `RSS Read` и фильтр `Limit`. Справа: дочерний ИИ-агент пишет саммари. | Давайте закрепим это на практике. Мы собираем новостной цикл: нода RSS Read собирает свежие статьи, а нода Limit оставляет только топ-5. Затем узел Loop по очереди передает каждый URL в дочерний воркфлоу. Там изолированный ИИ-агент читает текст и пишет краткую выжимку, возвращая ее обратно. Родительский процесс агрегирует эти 5 выжимок и публикует их единым постом. |
| **Сцена 6:** Схема Enterprise-архитектуры: Главный инстанс n8n -> Redis -> Воркеры (Task Runners). | Как только ваш контент-завод или CRM начнет генерировать тысячи задач в минуту, один сервер перестанет справляться. Здесь вступает в игру масштабирование: Queue Mode. В этом режиме главный инстанс n8n только маршрутизирует задачи. В качестве брокера сообщений выступает Redis, а независимые Task Runners — воркеры — извлекают задачи из очереди и выполняют их параллельно. Это горизонтальное масштабирование уровня Enterprise. |
| **Сцена 7:** Крупный план узла `Code`. Внутри пишется Python-скрипт, обрабатывающий `_input.all()`. | Визуальные узлы удобны, но иногда нужна грубая вычислительная мощь. Узел Code поддерживает выполнение Python-скриптов прямо внутри n8n. Вы можете обратиться ко всему массиву входных данных через объект `_input.all()` и использовать библиотеки Python для сложных математических или структурных трансформаций. Это идеальный мост между Low-Code и глубокой программной инженерией. |
| **Сцена 8:** Схема реагирования. Иконка `Webhook` вспыхивает, мгновенно запуская процесс без таймеров. | Переходим к паттернам триггеров. Запускать сценарии по таймеру — это прошлый век. Проектируйте событийно-ориентированные, Event-Driven архитектуры. Ваши автоматизации должны просыпаться только тогда, когда приходит Webhook от платежной системы или срабатывает Email Trigger по протоколу IMAP. Это разгружает сервера от пустых запросов и обеспечивает реакцию системы в реальном времени. |
| **Сцена 9:** Значок ошибки HTTP 429. Воркфлоу перенаправляет поток в узел `Wait` и использует настройки `Retry On Fail`. | При высоких нагрузках вы неизбежно столкнетесь с лимитами сторонних API — ошибками 429 Too Many Requests. Ваш процесс не должен падать. Настраивайте управление очередями с помощью параметров Retry On Fail на HTTP-нодах. Используйте узел Wait для реализации паттерна «Экспоненциальной задержки». А безнадежные задачи отправляйте в Dead Letter Queue для ручного разбора, чтобы основной конвейер продолжал работу. |
| **Сцена 10:** База данных PostgreSQL фиксирует статусы "processing" и "completed" между вызовами LLM. | И наконец, защита данных при сбое на середине цикла — Durable Execution. Если сервер перезагрузится, когда ИИ-агент обработал 500 из 1000 лидов, вы не должны начинать с нуля. Перед любым опасным вызовом API, сохраняйте промежуточное состояние задачи в базу данных (например, Postgres) со статусом «processing». При сбое скрипт возобновит работу строго с прерванной точки, гарантируя ACID-безопасность ваших транзакций. |
