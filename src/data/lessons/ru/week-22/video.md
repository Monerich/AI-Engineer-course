# Сценарий видео: Деплой и Надежное Выполнение

| Кадр / Scene | Звуковое сопровождение / Voiceover |
|:--- |:--- |
| **Scene 1:** Крупный план спикера, переход к анимированному Docker-контейнеру, оборачивающему Python-скрипты, зависимости и модели. | Добро пожаловать на 22-ю неделю: Деплой и Надежное Выполнение. Сегодня мы переводим ваших ИИ-агентов из локальных скриптов в продакшен. Начнем с Docker. Без него ваш код может отлично работать на ноутбуке, но упасть на сервере. Мы используем его, чтобы упаковать агента, зависимости и модели в единый, полностью воспроизводимый контейнер. |
| **Scene 2:** Разделенный экран: слева `docker-compose.yml`, справа архитектурная схема локальной сети (FastAPI, PostgreSQL, Redis). | Корпоративное приложение — это редко один скрипт. Во втором блоке мы используем Docker Compose для оркестрации нескольких контейнеров. Это позволяет безопасно настроить локальную маршрутизацию между вашим бэкендом, базой данных для сохранения состояний и очередями сообщений перед развертыванием в облаке. |
| **Scene 3:** Сетка из четырех колонок с логотипами Railway, Render, AWS и GCP. Акцент на скрытых переменных окружения (API-ключах). | Блок 3 посвящен релизу. Для быстрых прототипов платформы вроде Railway или Render предлагают деплой в один клик с поддержкой HTTPS. Для корпоративного масштаба используются AWS или GCP. И помните золотое правило безопасности: API-ключи всегда передаются через переменные окружения, никаких секретов в кодовой базе. |
| **Scene 4:** Динамический линейный график: резкий скачок HTTP-запросов вызывает автоматический запуск новых worker-узлов. | В блоке 4 мы боремся со скачками трафика. Система, работающая для десяти пользователей, рухнет при пяти тысячах. Мы настраиваем правила горизонтального автоскейлинга и архитектуру очередей, чтобы автоматически поднимать дублирующие stateless-воркеры при пиковых нагрузках, защищая серверы от перегрузок. |
| **Scene 5:** Схема шлюза, который перехватывает частый запрос и направляет его в Redis-кэш, минуя вызов LLM. | Блок 5 — это эффективность и стоимость. Агрессивное кэширование системных промптов и частых ответов может сэкономить вам до 90% затрат на API. Мы также настраиваем слой маршрутизации: простые задачи уходят быстрым моделям, а сложное планирование — флагманским. |
| **Scene 6:** Архитектура энтерпрайз-сервера с мигающими красными/зелеными индикаторами "Kill Switch" и алертами Sentry/CloudWatch. | Высокая доступность — фокус шестого блока. Мы проектируем отказоустойчивые архитектуры, которые переживают падения облака или LLM API. Внедрение автоматических circuit-breaker механизмов и рубильников (kill switch) позволяет мгновенно откатить систему при деградации метрик. |
| **Scene 7:** Чистый блок кода на Python, подсветка ключевых слов `async def`, `await` и эндпоинтов FastAPI. | Блок 7 объединяет ИИ с классическим бэкендом. Мы пишем масштабируемые API на FastAPI и Pydantic для строгой валидации данных. Используя асинхронные эндпоинты, мы гарантируем, что тяжелые и долгие вызовы нейросетей не заблокируют основной рабочий поток сервера. |
| **Scene 8:** Диаграмма LangGraph: контейнер падает из-за ошибки OOM, но мгновенно восстанавливает свое состояние из PostgreSQL. | В блоке 8 мы встречаем аппаратные сбои. Любой агент, работающий дольше 60 секунд, требует Durable Execution — надежного выполнения. Сохраняя состояние в базу данных после каждого шага (например, через PostgresSaver), мы делаем падение процесса полностью обратимым. |
| **Scene 9:** Логотипы Temporal и Inngest SDK поверх схемы, разделяющей детерминированный Workflow от хаотичных Activities. | Блок 9 вводит внешних оркестраторов. Inngest — самый простой путь к durability для Python, а Temporal превращает каждый вызов инструмента в надежный шаг. Они автоматически управляют экспоненциальными задержками и ретраями, если стороннее API падает. |
| **Scene 10:** Замок на базе данных, блокирующий дублирующийся API-запрос на списание кредитной карты при ретрае. | В блоке 10 мы применяем законы идемпотентности. Когда упавший контейнер перезапускается, мы обязаны предотвратить повторное выполнение необратимых действий. Мы внедряем ключи идемпотентности на каждый шаг и идемпотентную очистку временных файлов, гарантируя, что действия выполняются строго один раз. |
